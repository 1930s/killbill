Overview

Killbill maintains a reporting schema that is kept current in realtime for Analytics purposes. In contrary to other tables in the system, this set of tables is stable and potentially changes only between major revisions of Killbill. These tables are meant to be used as building blocks for reports and dashboards.

To distinguish them with Killbill objects, the Analytics objects are prefixed with the term Business, e.g. a Killbill account will have a representation in the Analytics subsystem by a Business Account object. For ease of tracking, Analytics objects contain id references to the Killbill ones but we strongly advise against using them for reporting purposes. Instead, use the external keys as much as possible, as they are tied to your business concepts. For instance, to count the number of accounts in the system, run a query using count(distinct account_key), not count(distinct account_id).

All of this information is also available in Kaui (in the analytics section), on a per account basis.

Accounts

The table bac (stands for Business Accounts) maintains information about Killbill accounts. The table doesn't contain historic records (it will contain at most one record per Killbill account).

BAC schema

Column name Column type Description
record_id int(11) unsigned Unique identifier for that record, e.g. 10
account_id char(36) Killbill account id, e.g. b2d69f1e-27cb-4c43-b997-71c28da4a170
account_key varchar(50) External key associated with that account
name varchar(100) Name associated with that account, e.g. Jean DuJardin
created_date bigint(20) Epoch (in UTC) when the account was created, e.g. 1341957285516
updated_date bigint(20) Epoch (in UTC) when the account was last updated, e.g. 1341957519902
balance decimal(10,4) Account balance in the account currency (takes into account invoices balance and credits), e.g. 254.4500
last_invoice_date bigint(20) Last invoice epoch (in UTC) for that account, e.g. 1341957522967
total_invoice_balance decimal(10,4) Sum of all invoices balance for that account in the account currency, e.g. 299.7500
last_payment_status varchar(255) Last payment status, e.g. SUCCESS
payment_method varchar(100) Default payment method (if any), e.g. Paypal
credit_card_type varchar(50) Default card type, if any, associated with that account, e.g. VISA
billing_address_country varchar(50) Billing country for that account, e.g. FR
currency varchar(50) Default account currency, e.g. MXN

Subscriptions

The table bst (stands for Business Subscription Transitions) maintains information about Killbill bundles and subscriptions. It describes the history of changes (both past and future) on a per subscription basis.

bst will contain all phase transitions for all subscriptions and can be used to build subscription timelines. A few remarks:

  • MIGRATE_BILLING and UNCANCEL subscription events are currently not stored in bst
  • The total_ordering column helps break ties for events with the same effective date: one can order chronologically the timeline by sorting records by nvl(next_start_date, requested_timestamp), total_ordering
  • prev_ fields will always be NULL for MIGRATE, ADD, RE_ADD, TRANSFER events
  • next_ fields will always be NULL for CANCEL and SYSTEM_CANCEL events
  • The requested_timestamp field reflects when the transition was requested, the next_start_date field reflects the effective transition time
  • The SYSTEM_CANCEL event is a special kind of event associated with a CANCEL event, used to get the time when a subscription is effectively cancelled. The SYSTEM_CANCEL requested_timestamp is the effective time when the subscription is cancelled, which is the effective time of the subscription CANCEL event

Invoices and invoice items

The tables bin (stands for Business Invoices) and bii (stands for Business Invoice Items) maintain information about invoices posted by account. The tables don't contain historic records (they will contain at most one record per Killbill invoice and invoice item).

BIN schema

Column name Column type Description
record_id int(11) unsigned Unique identifier for that record, e.g. 5
invoice_id char(36) Killbill invoice id, e.g. bb85b927-bfc9-48f4-8c72-2693733fabdd
invoice_number bigint(20) Invoice number, e.g. 401
created_date bigint(20) Epoch (in UTC) when the invoice was created, e.g. 1341958418930
updated_date bigint(20) Epoch (in UTC) when the invoice was last updated (should always be created_date as invoices are immutable), e.g. 1341958418930
account_id char(36) Killbill account id, e.g. 667e6a69-c8c2-4a04-a18e-31049818eb78
account_key varchar(50) Account external key
invoice_date date Invoice date in the account timezone, e.g. 2012-07-16
target_date date Invoice target date (maximum service period end date for the items) in the account timezone, e.g. 2012-07-16
currency char(5) Invoice currency, e.g. EUR
balance decimal(10,4) Invoice balance, e.g. 44.8500
amount_paid decimal(10,4) Amount paid, e.g. 0.0000
amount_charged decimal(10,4) Amount charged, e.g. 44.8500
amount_credited decimal(10,4) Amount credited, e.g. 2.1000

Payments

The table bip (stands for Business Invoice Payments) maintains information about Killbill payments, refunds and chargebacks. The table doesn't contain historic records (it will contain at most one record per Killbill invoice payment).

BIP schema

Column name Column type Description
record_id int(11) unsigned Unique identifier for that record, e.g. 1b43978b-68b5-4a05-8d51-18690148bf9e
payment_id char(36) Killbill payment id, e.g. 1b43978b-68b5-4a05-8d51-18690148bf9e
created_date bigint(20) Epoch (in UTC) when the associated invoice payment was created, e.g. 1342462966241
updated_date bigint(20) Epoch (in UTC) when the associated invoice payment was updated, e.g. 1342462966241

effective_date bigint(20) Epoch (in UTC) of the payment effective date, e.g. 1342462966241
amount decimal(10,4) Payment amount, e.g. 12.3400
currency char(5) Payment currency, e.g. USD
payment_error varchar(255) Gateway specific error message (for payment failures only), e.g. Insufficient funds
processing_status varchar(5) Killbill payment status, e.g. PAYMENT_FAILURE_ABORTED

Overdue statuses

The table bos (stands for Business Overdue Status) maintains information about Killbill overdue statuses. The table contains historic status for a given bundle.

BOS schema

Column name Column type Description
record_id int(11) unsigned Unique identifier for that record
bundle_id char(36) Killbill bundle id, e.g. ce3d45ec-30b7-4489-949c-d80808111993
external_key varchar(50) Bundle external key
account_key varchar(50) Account external key
status varchar(50) Overdue status, e.g. PRE_WARNING
start_date bigint(20) Epoch of the overdue status start period, e.g. 1342396800000
end_date bigint(20) Epoch of the overdue status end period, e.g. 1342396800000

Tags and custom fields

Information about tags and custom fields are captured in the tables bac_tags, bac_fields, bst_tags, bst_fields, bin_tags, bin_fields, bip_tags and bip_fields. The tables don't contain historic records (they will contain the current state of tags and custom fields).
